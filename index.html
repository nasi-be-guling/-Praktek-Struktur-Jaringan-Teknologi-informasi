Pertemuan 11

Logika "Di Balik Tombol Save/Simpan" & Konsep CRUD

Dibalik "Tombol Save/Simpan"

Yang Dilihat Perawat

Perawat di poli mengisi formulir pendaftaran di layar komputer (Front-End).

Setelah selesai, ia mengklik tombol biru besar bertuliskan "SIMPAN DATA". Tiba-tiba data tersimpan seolah-olah ada sihir (magic).

Yang Terjadi Sebenarnya

Tombol itu hanyalah pemicu (trigger). Saat diklik, aplikasi menerjemahkan isian formulir menjadi sebaris teks perintah rahasia.

Teks perintah inilah yang melesat melalui jaringan LAN menuju Server Database untuk dieksekusi.

Apa itu SQL?

SQL (Structured Query Language) adalah bahasa standar internasional yang digunakan aplikasi untuk "berbicara" dengan Database Relasional (MySQL, Oracle, dll).

Server Database tidak mengerti bahasa manusia atau klik mouse. Ia HANYA mengerti bahasa SQL.

Sebagai insinyur IT/Elektromedis, Anda harus bisa menulis bahasa ini secara manual untuk kebutuhan Troubleshooting.

Data Manipulation Language (DML) Dasar: C-R-U-D

Sebesar apapun aplikasi di dunia ini (SIMRS, e-Commerce), 99% fungsi database-nya bermuara pada 4 operasi fundamental yang disebut CRUD.

Create

Memasukkan baris data pasien baru.

Read

Membaca atau memfilter data ke layar.

Update

Mengedit data yang sudah ada.

Delete

Menghapus baris data dari sistem.

Pemetaan CRUD ke SQL

  • CREATE diterjemahkan menjadi INSERT INTO
  • READ diterjemahkan menjadi SELECT
  • UPDATE diterjemahkan menjadi UPDATE ... SET
  • DELETE diterjemahkan menjadi DELETE FROM

Best Practice

Bahasa SQL bersifat Case-Insensitive. Namun, perintah utamanya selalu ditulis dengan HURUF KAPITAL agar mudah dibedakan dengan nama tabel buatan kita.

1. CREATE (Insert Data)

Rumus Baku: Tentukan nama tabel, sebutkan kolomnya, lalu masukkan nilainya secara berurutan.

INSERT INTO pasien (no_rm, nama, gol_darah, umur) VALUES ('RM-001', 'Budi Santoso', 'O', 45);

Aturan Tipe Data

Nilai Teks (String) WAJIB diapit tanda kutip satu ('Budi'). Sedangkan nilai Angka TIDAK PERLU tanda kutip (45).

2. READ (Select Data)

Query yang paling sering digunakan untuk memanggil data dari gudang ke layar komputer.

Tampilkan SEMUA

Gunakan simbol bintang (asterisk *)

SELECT * FROM pasien;

Tampilkan Kolom SPESIFIK

Sebutkan nama kolom yang diinginkan

SELECT nama, gol_darah
FROM pasien;

Klausa WHERE (Filter)

Bagaimana jika tabel Anda memiliki 500.000 data, dan Dokter hanya mencari pasien bernama "Anton"? Menarik setengah juta data akan membuat komputer hang.

Kita harus menggunakan Saringan (Filter) yaitu klausa WHERE.

-- Mencari pasien bergolongan darah O SELECT nama, umur FROM pasien WHERE gol_darah = 'O';
-- Mencari pasien gol. darah A yang umurnya > 50 tahun SELECT * FROM pasien WHERE gol_darah = 'A' AND umur > 50;

3. UPDATE (Edit Data)

Rumus Baku: UPDATE (tabelnya) SET (kolom yang diubah) WHERE (kondisi targetnya).

UPDATE pasien SET umur = 46 WHERE no_rm = 'RM-001';
Apa yang terjadi jika Anda LUPA mengetikkan klausa WHERE pada perintah UPDATE?

SEMUA PASIEN DI RUMAH SAKIT AKAN BERUBAH UMURNYA MENJADI 46 TAHUN SECARA BERSAMAAN! DATA HANCUR!

4. DELETE (Hapus Data)

Membuang baris data secara permanen. Sama berbahayanya dengan UPDATE jika lupa memakai WHERE! Selalu gunakan Primary Key (No_RM) sebagai sasaran tembak.

DELETE FROM pasien WHERE no_rm = 'RM-002';

Best Practice Rumah Sakit

Di dunia medis, data haram di-DELETE permanen (Hard Delete) karena alasan hukum. Programmer biasanya memakai "Soft Delete" (UPDATE is_active = 0) agar data hilang dari layar tapi tetap utuh di database.

Pertemuan 12

Advanced SQL: Relasi Tabel & JOIN Query

Irelevansi Tabel Raksasa (Flat Table)

Pemula sering membayangkan Database RS seperti satu file Excel raksasa yang berisi semua riwayat kunjungan pasien dalam satu baris panjang:

No_KunjunganNama PasienAlamatNama DokterSpesialisDiagnosa
K-001BudiJl. Merdekadr. AndiPenyakit DalamTifus
K-002SitiJl. Mawardr. AndiPenyakit DalamFlu
K-003BudiJl. Merdekadr. BudiBedahUsus Buntu

Sistem "Tabel Datar" ini SANGAT BURUK dan DILARANG KERAS di industri!

Mengapa Tabel Raksasa Berbahaya?

  • Redundansi Data (Pemborosan): Nama "Budi" dan alamatnya diketik berulang-ulang setiap ia berobat. Ini memboroskan server.
  • Ancaman Inkonsistensi (Typo): Bagaimana jika pada kunjungan ke-3 perawat salah ketik menjadi "Buddi"? Sistem mengira itu orang yang berbeda.
  • Kesulitan Update: Jika Budi pindah alamat, Anda harus meng-update ratusan baris riwayat kunjungannya satu per satu.

Solusinya: Pisahkan Tabelnya!

RDBMS (Relational Database) memecahnya menjadi tabel-tabel spesifik. Di RS, minimal butuh 3 tabel inti:

Tabel Pasien (Master)

Hanya menyimpan data biodata permanen. Satu orang = Satu Baris.

Tabel Dokter (Master)

Hanya menyimpan data spesialisasi dokter yang bekerja di RS.

Tabel Kunjungan (Transaksi)

Tabel yang mencatat kejadian/peristiwa pertemuan pasien dan dokter. Di sinilah keajaiban relasi terjadi.

Kunci Utama vs Kunci Tamu

Bagaimana Tabel Kunjungan tahu pasien mana yang diperiksa tanpa mengetik ulang namanya?

Primary Key (PK)

Identitas unik mutlak di tabel asalnya. Contoh: no_rm di Tabel Pasien.

Foreign Key (FK)

Kunci Tamu. Primary Key dari tabel lain yang "dipinjam" dan disisipkan sebagai "tali pengikat".

Di Tabel Kunjungan, kita TIDAK menulis nama pasien. Kita HANYA menulis no_rm-nya saja!

Arsitektur 3 Tabel (Smart Clinic)

1. Tabel Pasien:
no_rm (PK), nama_pasien, alamat

2. Tabel Dokter:
id_dokter (PK), nama_dokter

3. Tabel Kunjungan:
id_kunjungan (PK)
tanggal
no_rm (FK -> Pasien)
id_dokter (FK -> Dokter)
diagnosa

Masalah Tampilan Transaksi

Jika kita melakukan SELECT * FROM kunjungan;, hasilnya akan terlihat aneh bagi manusia biasa:

id_kunjungantanggalno_rmid_dokterdiagnosa
K-00112 JanRM-10DR-01Tifus
K-00215 JanRM-20DR-01Flu

Dokter atau Kasir RS akan marah jika melihat layar ini. Mereka tidak hafal siapa "RM-10" dan siapa "DR-01". Mereka butuh melihat NAMA aslinya!

INNER JOIN: Advanced SQL

Untuk menyatukan kembali data yang sudah kita pisah-pisah secara fisik tersebut (saat akan ditampilkan ke layar komputer), kita menggunakan perintah JOIN.

INNER JOIN akan mencocokkan baris dari Tabel A dan Tabel B berdasarkan "Tali Pengikat" (Primary Key dan Foreign Key) yang sama nilainya.

Logika JOIN

"Tampilkan data dari tabel Kunjungan, lalu gabungkan dengan tabel Pasien, DI MANA no_rm di tabel kunjungan SAMA DENGAN no_rm di tabel pasien."

Sintaks JOIN (2 Tabel)

Menampilkan ID Kunjungan, Tanggal, dan NAMA Pasien.

SELECT kunjungan.id_kunjungan, kunjungan.tanggal, pasien.nama_pasien FROM kunjungan INNER JOIN pasien ON kunjungan.no_rm = pasien.no_rm;

Perhatikan: Kita harus menyebutkan nama tabelnya di depan nama kolom (Tabel.Kolom) agar SQL tidak bingung jika ada nama kolom yang sama di kedua tabel.

Sintaks JOIN (3 Tabel Sekaligus)

Menampilkan riwayat lengkap: Tanggal, Nama Pasien, Nama Dokter, dan Diagnosa.

SELECT k.tanggal, p.nama_pasien, d.nama_dokter, k.diagnosa FROM kunjungan AS k INNER JOIN pasien AS p ON k.no_rm = p.no_rm INNER JOIN dokter AS d ON k.id_dokter = d.id_dokter;

*Aliases: Kita menggunakan Alias (AS k, AS p, AS d) untuk menyingkat nama tabel yang panjang agar kode lebih bersih.*

Fungsi Agregasi Dasar

Bagaimana Direktur RS melihat angka statistik "Total Pasien Hari Ini" di layar Dashboard-nya? Sistem menggunakan fungsi agregasi seperti COUNT().

-- Menghitung total semua pasien terdaftar SELECT COUNT(*) FROM pasien;
-- Menghitung pasien yang bergolongan darah O saja SELECT COUNT(*) FROM pasien WHERE gol_darah = 'O';

Skenario Praktikum (Lab XAMPP)

  • Persiapan: Buka localhost/phpmyadmin. Buat Database baru sesuai aturan NIM.
  • Pembuatan Tabel: Buat tabel master (pasien, dokter) dan tabel transaksi (kunjungan) dengan Primary Key & Foreign Key.
  • Aksi CRUD: Eksekusi perintah INSERT, SELECT, UPDATE, dan DELETE murni menggunakan tab SQL.
  • Uji Query DML JOIN: Tunjukkan kemampuan Anda merelasikan 3 tabel menggunakan INNER JOIN.
